Architectures web actuelles

Architecture classique

  • Existe depuis le début du web
  • Affichage de documents statiques
  • Le serveur s'occupe de tout : gestion des données, fonctions métier, génération de l'IHM
  • Le client ne fait qu'afficher le document et les ressources statiques que le serveur lui envoie
  • Archi adaptée aux machines peu puissantes de l'époque, et au fait que le web n'avait vocation qu'à faire circuler de la connaissance

Architecture classique

  • Simple à mettre en place
  • Affichage de documents statiques
  • Rechargement de la page à chaque action utilisateur
  • Ancienne mais toujours adaptée pour certains besoins
  • La page refresh à chaque action
  • Même si il y a un peu d'interactivité, ça reste limité

Architecture AJAX

  • Le client reçoit toujours l'IHM générée par le serveur
  • Mais grâce à XHR et DOM, il est capable de mettre à jour des portions de pages sans tout recharger
  • Le serveur peut alors renvoyer soit des données brutes, soit des portions d'IHM
  • Accompagne les évolutions des usages du web : réseaux sociaux, interactivité, participation...
  • Accompagne aussi les évolutions des clients, capable d'en faire un peu plus qu'avant

Architecture AJAX

  • Utilisation de nouvelles technologies JavaScript : XHR et DOM
  • Meilleures interactivité et réactivité
  • Plus complexe que l'architecture classique
  • Des librairies JS sont apparues pour faciliter les développements
  • Mais c'est resté basé sur du code issu de l'architecture classique
  • Donc la complexité a très vite augmenté (exemple Facebook)
  • Des informations sont ajoutées à la page pendant la saisie
  • Mais on refresh quand même quand on passe à un autre contexte

Architectures actuelles – problématiques

  • Très peu d'interactivité OU trop de complexité
  • Essor des applications natives : nouvelles attentes
  • Enjeux majeurs pour une entreprise du web

Nous avons besoin de concevoir les sites web différemment

  • Les utilisateurs sont habituées au comportement des applications natives
  • Ils veulent retrouver ça quand ils naviguent sur le web
  • Il faut donc leur proposer ce qu'ils attendent, c'est stratégique pour les entreprises du web
  • L'architecture AJAX peut potentiellement répondre à ce besoin, mais au prix d'une complexité qui ne fera qu'augmenter
  • Il faut donc une autre architecture

Nouvelle architecture :

MV* côté client

Architecture MV* côté client

  • Le serveur ne s'occupe plus du tout de l'IHM, c'est le client qui s'en charge
  • Chaque partie a donc un rôle bien précis : présentation pour le client, données pour le serveur
  • On ne refresh jamais
  • Les données sont récupérée sur le serveur quand on en a besoin
  • Et le navigateur s'occupe de générer et mettre à jour l'IHM
  • Client et serveur sont totalement indépendant, ils discutent via HTTP

Architecture MV* côté client – utilisation massive de JavaScript

(plus le score est faible, plus les performances sont élevées)

  • Cest possible grâce aux efforts des éditeurs de navigateurs

Architecture MV* côté client – utilisation massive de JavaScript

  • Avant JS était un langage jouet
  • Aujourd'hui un écosystème existe
  • Et on est capable d'industrialiser des développements JS

Architecture MV* côté client – back-end autonome

  • L'api répond aux requêtes qu'on lui envoie, avec les données demandées (ou une erreur...)

Architecture MV* côté client – back-end autonome

  • L'API étant totalement déconnectée du client, elle peut servir un client web, mais aussi des applications mobiles...
  • Le support et le format d'échange sont standardisés
